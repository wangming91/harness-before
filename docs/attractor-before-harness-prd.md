# Attractor Before Harness PRD

## 1. 产品名称

Attractor Before Harness

## 2. 产品定位

面向 AI 参与的大规模开发场景的轨迹收敛框架。它通过“吸引子 + 计划 + 验证 + 审计 + 记忆”的组合，把系统应该向哪里收敛这件事显式化，并让后续所有开发活动都围绕这个方向展开。

## 3. 问题陈述

现有 AI 开发流程通常只关注单次状态是否正确，缺少对长期结构演化的约束。结果是：

- 局部代码看起来合理，但整体架构持续漂移。
- 生成和验收在同一上下文内完成，容易形成自我验证。
- 历史上的失败经验无法被系统性复用。
- 计划、测试、审计之间缺少统一的收口逻辑。

## 4. 产品目标

1. 把系统长期收敛方向显式化。
2. 把计划、验证、审计、记忆串成闭环。
3. 把完成判定从实现上下文中剥离出来。
4. 让仓库成为系统真相来源。
5. 让失败经验可检索、可传承、可复用。

## 5. 目标用户

### 5.1 架构负责人

负责定义系统吸引子、边界和演进原则。

### 5.2 AI 协作者

围绕既定吸引子生成代码、文档、测试和重构建议。

### 5.3 验收/审计人员

独立判断当前任务是否真的完成，是否满足收口条件。

### 5.4 维护者

沉淀失败路径、反例和被推翻的假设。

## 6. 用户场景

### 场景 1：新模块开发前先定方向

用户准备新增一个模块，但担心 AI 直接开始写代码后把系统带偏。

需求：

- 先读取当前吸引子文档。
- 先确认这个模块的职责边界和依赖方向。
- 再生成计划和验证要求。

验收标准：

- 系统必须在生成前显示该模块绑定的吸引子版本。
- 必须明确列出 goals / non-goals。
- 必须给出 exit criteria 和 closure evidence。
- 未绑定吸引子的计划不能进入执行状态。

### 场景 2：AI 完成了一轮实现，但需要独立判断是否真的完成

用户看到验证通过，但怀疑只是表面正确。

需求：

- 触发独立审计流程。
- 审计必须使用 fresh session 或独立上下文。
- 审计不能依赖实现者的完成声明。

验收标准：

- 审计报告必须引用仓库中的外部证据。
- 审计结果必须给出 pass / fail / partial。
- 若证据不足，必须返回 need_info 或 fail。
- 不能在无审计结果的情况下关闭计划。

### 场景 3：同一问题反复出现，需要沉淀经验

用户发现某类失败反复发生，例如测试脆弱、边界漂移、术语混乱。

需求：

- 将问题写入 memory。
- 关联到相关 plan、audit 和 evidence。
- 下次相似问题出现时可以检索到历史结论。

验收标准：

- memory 条目必须包含类型、摘要、证据和关联对象。
- 检索结果必须支持按问题类型和关键词命中。
- 已被证伪的前提必须能在后续任务中被引用。

### 场景 4：吸引子需要演进

用户发现旧的结构基线已经无法支撑新的演进方向。

需求：

- 提出新的吸引子版本。
- 记录与旧版本的差异。
- 保留迁移原因和废弃范围。

验收标准：

- 新吸引子必须有版本号。
- 必须记录 superseded 的旧版本。
- 必须给出变更理由、兼容影响和适用范围。

### 场景 5：计划被验证失败阻断

用户的计划在 lint、测试或审计阶段失败。

需求：

- 明确失败分类。
- 明确是继续修复、拆分计划，还是回滚。
- 将失败证据沉淀下来。

验收标准：

- 验证失败必须归类到具体检查项。
- 计划状态必须能进入 blocked。
- blocked 状态必须附带下一步动作。

## 7. 功能需求

### 7.1 吸引子管理

- 创建、查看、更新、废弃吸引子。
- 支持版本化。
- 支持作用域定义：repo / package / module。
- 支持高阶不变量、依赖规则、边界规则。

### 7.2 计划管理

- 创建计划并绑定吸引子。
- 定义 baseline、goals、non-goals、exit criteria。
- 跟踪任务拆分和状态变化。
- 记录 closure evidence。

### 7.3 验证管理

- 支持执行 lint、typecheck、build、test 等检查。
- 统一结果格式。
- 记录失败项和产物。

### 7.4 审计管理

- 支持独立审计请求。
- 支持审计结论记录。
- 支持审计失败后的追踪动作。

### 7.5 记忆管理

- 支持保存失败经验。
- 支持保存被推翻的假设。
- 支持基于上下文检索历史结论。

### 7.6 路由与导航

- 根据问题类型推荐读取顺序。
- 区分 baseline、plan、audit、history。

## 8. 非功能需求

- 可追溯：所有关键结论必须能回到证据。
- 可审计：每次关闭必须有独立审计。
- 可版本化：吸引子、计划和记忆都必须可 diff。
- 可扩展：支持多仓库、多团队、多语言。
- 可集成：能接 CI、issue、log、测试框架。

## 9. 约束

- 不允许把验证结果直接当作最终完成结论。
- 不允许同一上下文既实现又审计。
- 不允许没有吸引子的计划进入执行。
- 不允许关闭缺少证据链的任务。

## 10. 验收标准总表

### 10.1 基础能力验收

- 能创建并版本化吸引子。
- 能创建计划并绑定吸引子。
- 能执行验证并记录结果。
- 能触发独立审计。
- 能记录 memory。

### 10.2 过程能力验收

- 任务必须有 baseline。
- 任务必须有 exit criteria。
- 任务必须有 closure evidence。
- 审计必须独立于实现上下文。
- 关闭必须由审计结果驱动。

### 10.3 结果能力验收

- 能区分局部正确和整体收敛。
- 能识别假完成。
- 能识别结构漂移。
- 能复用历史失败经验。

## 11. 成功指标

- 计划关闭后回滚率下降。
- 审计驳回率可见且可解释。
- 重复失败路径下降。
- memory 命中率上升。
- 任务平均关闭轮次下降。

## 12. 范围边界

### 12.1 本期做

- 吸引子、计划、验证、审计、记忆的最小闭环。
- Git-native 文档和元数据。
- 本地可执行的基础流程。

### 12.2 本期不做

- 不做复杂流程编排平台。
- 不做重型图形化编辑器。
- 不做完整的任务管理替代品。
- 不做通用知识库。

## 13. 结论

这个产品不是“更严的 harness”，而是把“系统应向哪里收敛”变成第一类工程对象。
