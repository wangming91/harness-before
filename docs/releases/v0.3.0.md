# ABH v0.3.0 Release Notes

Release date: 2026-05-24

## Summary

v0.3.0 marks the completion of Stage 3: Verify Runner. ABH can now move beyond recording human validation results and can execute a plan's validation checklist locally, persist machine-readable evidence, classify failures, and expose trust/freshness signals for audit review.

## Included Milestones

- Stage 1 governance foundation: ABH object schema versions, `abh doctor`, CI baseline, editable install path, and roadmap/task-board synchronization.
- Stage 2 Agent Protocol Foundation: explicit JSON CLI envelope, structured ABH errors, read-only MCP stdio tools, and controlled MCP write tools guarded by `confirm=true`.
- Stage 3 Verify Runner: `abh verify run <plan>`, `abh plan update <plan>`, environment metadata, `trust_level`, stale summary, failure classifications, atomic ABH writes, and focused domain module boundaries.

## Key Capabilities

- `abh verify run <plan_id>` executes validation checklist commands and records command, exit code, duration, stdout/stderr artifact text, trust level, environment metadata, and failure classifications.
- `abh plan status --json` exposes latest verification summary, including trust level and stale reasons.
- `abh plan update <plan_id>` appends goals, non-goals, exit criteria, validation checklist items, and closure evidence with deduplication and JSON/Markdown dual-write.
- ABH JSON and Markdown save paths use single-file atomic writes with local file locking.
- `abh.core` now re-exports focused domain modules for plans, audits, verifications, memory, drift, routing, errors, and storage compatibility.

## Known Limitations

- `verify run` executes local shell commands. It records trust and environment metadata, but it is not an isolated runner or tamper-proof proof.
- There is no CI-backed verification runner yet. CI exists for repository checks, but verification records are still local-shell evidence.
- Stale verification summaries are audit signals and do not automatically block plan close.
- Drift analysis remains keyword/rule based.
- Attractor Registry is not included in v0.3.0; it starts in Stage 4 after this release-prep slice.

## Upgrade Notes

- Package metadata and runtime version are now `0.3.0`.
- Existing ABH JSON records remain schema-version compatible.
- Existing CLI and MCP contracts remain compatible with v0.2.0 behavior, with added Stage 3 verification metadata surfaces.
